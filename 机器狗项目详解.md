## 1. 项目总览

### 1.1 项目定位

一款基于 **STM32F407ZE** 主控的**四足仿生机器狗**，通过 **PCA9685** 驱动 8 路舵机完成肢体动作，并支持 **蓝牙网页遥控** 和 **ESP32 语音识别** 两种交互方式。

### 1.2 核心技术点（一句话总结）

| 模块 | 核心点 |
|------|--------|
| 主控 | STM32F407ZE，168MHz Cortex-M4 |
| 舵机驱动 | PCA9685 (I2C + 16路12位PWM) |
| 姿态感知 | MPU6050 (I2C 6轴陀螺仪+加速度计) |
| 无线交互 | 蓝牙模块（网页遥控）+ ESP32（语音） |
| 状态反馈 | WS2812 地址可控 RGB LED |
| 关键算法 | 运动学逆解 + 步进平滑插值 |
| 软件架构 | 事件驱动：串口指令 → 动作解析 → 舵机控制 |

### 1.3 系统整体框图

```
┌──────────────────────────────────────────────────────┐
│                   用户交互层                          │
│   网页遥控（手机）         语音指令（麦克风）         │
└──────┬───────────────────────────┬───────────────────┘
```

## 2. 硬件架构详解

### 2.1 主控：STM32F407ZE（躯干）

| 参数 | 值 |
|------|-----|
| 内核 | ARM Cortex-M4（带 FPU、DSP 指令） |
| 主频 | 168 MHz |
| Flash / RAM | 典型 512KB~1MB Flash / 192KB RAM |
| 集成外设 | UART、SPI、I2C、CAN、USB、定时器等 |

**时钟电路**：
- 主时钟：8MHz 石英晶振 + 两颗 22pF 负载电容，PLL 倍频到 168MHz。
- RTC 时钟：32.768 kHz + 15pF 电容，支持低功耗模式。

**复位电路**：
- 10kΩ 上拉 + 0.1μF 电容组成 RC 复位，按键 SW1 手动复位。
- NRST 串联 1MΩ 电阻提升抗干扰。

**电源滤波**：每个 VDD 引脚就近放 104（0.1μF）陶瓷电容，VSSA/VDDA 侧放 10μF 钽电容。

**为什么选 F407？**
- 168MHz 主频足够算实时逆解 + PWM 调度。
- 带 FPU，运动学计算用到 `float` 开销小。
- 外设丰富，4 路 UART + 多路 I2C/SPI 可一次性把蓝牙、ESP32、PCA9685、MPU6050 全接上。

---

### 2.2 次控：ESP32（狗头）

**职责**：语音识别前置处理。麦克风采集 → 识别指令 → 通过 UART4 发给 STM32。

**为什么把语音放在 ESP32 而不是 STM32？**
- ESP32 双核，主频 240MHz，Wi-Fi/蓝牙内置，便于云端语音识别或本地离线识别。
- STM32 专注实时运动控制，解耦后系统更稳定。
- 典型 IoT 分工思路：感知前置在边缘节点，控制后置在主控。

---

### 2.3 舵机驱动：PCA9685（关键核心）

| 参数 | 值 |
|------|-----|
| 通信 | I2C（7位地址，默认 0x40） |
| PWM 通道 | 16 路 |
| PWM 分辨率 | 12 位（0~4095） |
| 输出频率 | 24 Hz ~ 1526 Hz 可调，舵机常用 50Hz |

**工作原理**：
- 内部有一个独立振荡器（默认 25MHz），通过 **prescale（预分频器）** 产生 PWM 基频。
- 每个通道有 4 个寄存器：`ON_L / ON_H / OFF_L / OFF_H`，定义占空比的起始和结束时刻。
- 舵机控制：50Hz 周期 = 20ms，对应 4096 步；0.5ms~2.5ms 脉宽对应 0°~180°。

**为什么用 PCA9685 而不是用 STM32 的定时器直接输出 PWM？**
- 机器狗需要至少 8 路 PWM（4 腿 × 2 关节），STM32 的 PWM 通道够用但占用大量定时器资源。
- PCA9685 通过 I2C 扩展 16 路，节省 MCU 资源、布线简单。
- 所有通道共享同一基频，舵机同步性更好。

---

### 2.4 姿态传感器：MPU6050

**功能**：6 轴（3 轴加速度 + 3 轴陀螺仪），通过 I2C 读取。

**在本项目中的作用**：
- 判断机器狗是否倾倒。
- 可用于闭环姿态调整（如果做进阶）。
- 提供数据给上位机做姿态可视化。

---

### 2.5 状态灯：WS2812

**特性**：地址可控 RGB LED，单总线控制，串联级联。

**控制方式**：通过 GPIO 翻转生成精确时序：
- Bit 0：高电平 220~380ns + 低电平 580ns~1.6μs
- Bit 1：高电平 580ns~1.6μs + 低电平 220ns~420ns
- 复位：低电平 > 50μs

**数据顺序**：**GRB**（不是 RGB！），每灯 24bit，最高位先发。

**为什么时序要求这么苛刻？**
- WS2812 内部解码完全依赖脉宽判断 0/1，偏差超过阈值就解码错误。
- 因此一般用**汇编精确计数** 或 **SPI/DMA 字节法** 或 **PWM+DMA** 控制。

---

### 2.6 电源管理

| 电压 | 芯片 | 供电对象 |
|------|------|----------|
| 7.3V | 锂电池原始输出 | 总输入 |
| 5V / 3A | LM2596S | 舵机 |
| 3.3V | AMS1117-3.3V | STM32、ESP32、传感器 |

**关键设计点**：舵机功耗大（一次动可能 1A+），必须独立供电通路，避免 MCU 掉电复位。

## 4. 核心算法深度解析

### 4.1 运动学逆解（InverseResolve）

**是什么？**
已知足尖目标位置 `(x, y)`，反推大腿舵机角度 `θ1` 和小腿舵机角度 `θ2`。

**几何模型**（二连杆平面腿）：

```
       关节A (髋)
        ●──────────┐
         \         │ L1（大腿长度）
          \  θ1    │
           \       │
            ● 关节B (膝)
             \
              \ L2（小腿长度）
               \
                ● 足尖 (x, y)
```

设：大腿长 L1，小腿长 L2，大腿与竖直方向夹角为 θ1，小腿相对大腿的内夹角为 θ2。

**推导思路**（余弦定理）：

第一步：足尖到髋关节的距离
```
d = √(x² + y²)
```

第二步：用余弦定理求膝关节角 θ2
三角形三边为 L1、L2、d：
```
cos(π - θ2) = (L1² + L2² - d²) / (2·L1·L2)
θ2 = π - arccos((L1² + L2² - d²) / (2·L1·L2))
```

**可达性判断**：若 `d > L1 + L2` 或 `d < |L1 - L2|`，目标点不可达，函数返回 `-1`。

第三步：求大腿角 θ1
```
α = atan2(x, y)                    // 足尖相对髋关节的方位
β = arccos((L1² + d² - L2²) / (2·L1·d))  // 大腿与足尖连线的夹角
θ1 = α - β
```

**特殊情况**：x=0（足尖正下方），α 退化，需单独处理避免除零。

**在代码中的作用**：把"我想让脚落在哪个位置"转换成"舵机要转多少度"，这是控制四足机器人的通用思路。

---

### 4.2 AngleMove 平滑过渡

**问题**：如果直接 `setAngle(90°)`，舵机会以最大速度冲到目标，机械冲击大、易损伤、动作生硬。

**解法**：分步逼近目标角度。

```c
void AngleMove(uint8_t ch, float target, float step) {
    float current = g_currentAngle[ch];
    if (fabs(target - current) < step) {
        PCA9685_setAngle(ch, target);
        g_currentAngle[ch] = target;
    } else {
        current += (target > current) ? step : -step;
        PCA9685_setAngle(ch, current);
        g_currentAngle[ch] = current;
    }
}
```

**每次只走一小步**（如 1°），配合动作节拍的 delay，得到平滑运动轨迹。

**可进一步优化**：改成 S 曲线插值（加速度连续），动作更自然。

### 4.3 动作序列执行（DogAction）

**思路**：把"前进一步"拆解成若干关键帧（腿1抬→腿1前伸→腿1落→腿3抬→...），每一帧包含 4 条腿的坐标 + 延时。

```c
// 伪代码示意
const char* ActionList_Run[] = {
    "A:(20,50) B:(-20,50) C:(-20,50) D:(20,50) T:100",  // 关键帧1
    "A:(20,40) B:(-20,50) C:(-20,50) D:(20,50) T:80",   // 关键帧2
    // ...
    NULL
};

void DogAction(const char** list) {
    for (int i = 0; list[i] != NULL; i++) {
        int delay = XYLineToAngel(list[i]);  // 解析并下发
        HAL_Delay(delay);
    }
}
```

**函数族**：`DogRun / DogBack / DogLeft / DogRight / DogUp / DogDown / DogSquat / DogBow / DogWave`，每个都是一段关键帧序列。

### 4.4 WS2812 时序控制

**思路 A：裸 GPIO 延时**（代码中的方式）
```c
void ws2812_WriteBit1(void) {
    GPIO_HIGH();
    // NOP 填充达到 ~800ns
    GPIO_LOW();
    // NOP 填充达到 ~400ns
}
```

**思路 B：SPI + DMA**（更稳更推荐）
- 把每个 WS2812 bit 用 3~4 个 SPI bit 编码（比如 `110` 代表 1，`100` 代表 0）。
- DMA 一次性推送，CPU 零占用。

## 5. 通信协议原理与实现

### 5.1 I2C

**基础特性**：
- 两线：SCL（时钟）+ SDA（数据）
- 半双工，主从架构，支持多主多从
- 典型速率：100kHz（标准）、400kHz（快速）
- 每设备有 7 位地址

**通信流程**：START → 设备地址+读/写位 → ACK → 寄存器地址 → ACK → 数据 → ACK → STOP

**本项目中 I2C 总线上挂了谁？**
- PCA9685（0x40）
- MPU6050（0x68）
- 可扩展 AHT20、OLED 等

**为什么 I2C 地址不会冲突？** 每个芯片默认地址不同，部分可通过 ADDR 引脚配置偏移。

---

### 5.2 UART

**特性**：
- 异步，全双工，点对点
- 常用波特率：9600 / 115200
- 无时钟线，依赖双方约定波特率

**本项目帧格式**：ASCII 文本指令，如 `"FORWARD"`、`"LEFT"`、`"LED RED ON"`。
- **接收方式**：`HAL_UARTEx_ReceiveToIdle` —— 接收到空闲帧（一段时间无数据）自动触发回调，避免指令长度不定。

**弱点**：ASCII 易读但抗干扰差。**升级方向**：二进制协议 + CRC 校验，如 `0xAA [CMD] [LEN] [DATA...] [CRC]`。

---

### 5.3 PWM

**核心概念**：
- 周期 T = 1 / 频率
- 占空比 = 高电平时间 / 周期
- 舵机：50Hz (20ms 周期)，脉宽 0.5ms~2.5ms 对应 0°~180°

**PCA9685 的 PWM 分辨率**：
```
每步时间 = 20ms / 4096 ≈ 4.88μs
0° 对应 OFF 值 ≈ 0.5ms / 4.88μs ≈ 102
180° 对应 OFF 值 ≈ 2.5ms / 4.88μs ≈ 512
```
所以角度到 OFF 值的映射：`off = 102 + angle * (512 - 102) / 180`

## 6. 引脚分配与电路要点

### 6.1 主要引脚分配

| 功能 | 引脚 | 备注 |
|------|------|------|
| USART1 TX/RX | PA9 / PA10 | 调试串口 |
| USART2 TX/RX | PA2 / PA3 | 蓝牙（脚位据板子微调） |
| UART4 TX/RX | PC10 / PC11 | ESP32 语音 |
| I2C1 SCL/SDA | PB10 / PB11 | PCA9685 + MPU6050 |
| 按键 | PF5~PF9 | 上拉，按下拉低 |
| TIM2 PWM | — | RGB LED 呼吸 |
| TIM5 PWM | — | 蜂鸣器 |
| WS2812 DIN | GPIO | 精确时序 |

### 6.2 电路要点清单

- 晶振走线尽量短，远离高频信号线，防止时钟抖动。
- NRST 引脚串电阻+上拉，避免复位误触发。
- VDD 引脚每处就近加 0.1μF 去耦电容。
- 舵机共地但独立 5V 电源线，避免大电流通过 MCU 地。
- I2C 总线必须加 4.7kΩ 上拉电阻（若板子没内置，要外接）。

## 7. 动作控制完整链路

以**"前进"一次为例**：

```
用户点击网页"前进" 按钮
        ↓
手机 → HC-05 蓝牙模块（蓝牙协议）
        ↓
HC-05 → STM32 USART2 （串口字节）
        ↓
USART2 RX 中断 → 填充缓冲区
        ↓
空闲中断触发 → ParseCommand("FORWARD")
        ↓
调用 DogRun()
        ↓
遍历 ActionList_Run[]
        ↓
每条关键帧 XYLineToAngel:
   解析 4 腿坐标 (x, y)
        ↓
   InverseResolve(x, y, &θ1, &θ2)
        ↓
   SetAngelByID(leg_id, θ1, θ2)
        ↓
   PCA9685_setAngle(ch, angle) × 8
        ↓
   PCA9685 I2C 写入寄存器
        ↓
   16 路 PWM 输出
        ↓
   舵机旋转到目标角度
        ↓
   HAL_Delay(帧延时)
        ↓
继续下一帧 → 直到走完一步
```

## 8. 常见问题与解决方案

| 问题 | 现象 | 原因分析 | 解决方案 |
|------|------|----------|----------|
| 串口不显示 | 插上 USB 无日志输出 | 驱动未识别，或未供电唤醒 | 插电稍等 5~10s；或接入电池后再连 USB |
| 舵机姿态异常 | 四腿角度乱、机身歪斜 | 舵机安装时零位不一致 | 标定偏置：目标角度 − 基准角度，存到 `Px[]` 数组 |
| 指令无响应 | 蓝牙/语音指令发了没反应 | 狗头 IO 接线错误 | 严格按指导书接线，逐根万用表量 |
| WiFi 控制页打不开 | 手机访问 192.168.4.1 失败 | 手机数据开启，系统偏好走流量 | 关掉移动数据和 Wi-Fi 自动切换 |
| 呼吸灯卡顿 | 主循环延迟动作变慢 | 使用了阻塞式 `HAL_Delay` | 改用 TIM 中断非阻塞更新 PWM |
| 舵机过行程 | 机械结构碰撞损坏 | 角度无边界检查 | `if (angle > 180) angle = 180;` 限幅 |
| I2C 读不到 MPU6050 | `HAL_I2C_IsDeviceReady` 返回错误 | 地址错/上拉缺失/接线错 | 检查 0x68 地址、上拉电阻、SDA/SCL 是否接反 |

## 9. 面试高频问答

### Q1：为什么选 PCA9685？STM32 直接输出 PWM 不行吗？

> 可以，但 STM32 每路 PWM 占用一个定时器通道，16 路会吃掉大半定时器资源。PCA9685 通过 I2C 单总线扩展，只需 2 根线就能驱动 16 路舵机，节省 MCU 引脚和外设资源，同时所有通道共享同一基频，舵机动作同步性更好。

### Q2：I2C 和 SPI 的区别？为什么这里选 I2C？

> - **I2C**：两线（SCL+SDA）、半双工、多设备共线、速率低（标准 100k、快速 400k）、需要地址。
> - **SPI**：四线（SCK+MOSI+MISO+CS）、全双工、速率高（MHz 级）、每设备独立片选。
>
> 本项目舵机驱动只需要写命令，对速率要求不高，且 PCA9685 + MPU6050 可以共用一条总线，I2C 是最佳选择。

### Q3：运动学逆解是什么？怎么实现的？

> 根据足尖目标位置反推关节角度。我这里是二连杆模型，用**余弦定理**：已知大腿长 L1、小腿长 L2 和足尖到髋关节的距离 d，可以算出膝关节夹角；再用 `atan2` 得到大腿与竖直方向的夹角。最后加上角度偏置（标定值）送给 PCA9685。

### Q4：为什么做舵机平滑过渡？

> 如果直接下发目标角度，舵机会以最大速度冲过去，产生机械冲击，时间长了损伤结构件，动作看着也很生硬。我用 `AngleMove` 函数把目标角度拆成多个小步长，每个周期走一小步，整体动作平滑可控。

### Q5：PWM 控制舵机的原理？

> 舵机用 50Hz PWM（周期 20ms），内部电路解析脉宽：
> - 0.5ms 对应 0°
> - 1.5ms 对应 90°
> - 2.5ms 对应 180°
>
> 脉宽和角度是线性关系，所以给 PCA9685 的 OFF 寄存器写对应值即可。

### Q6：WS2812 数据顺序？时序要求？

> 数据顺序是 **GRB**，每灯 24bit。Bit 0 和 Bit 1 靠高低电平持续时间区分（大约 350ns 高 + 800ns 低 为 0；800ns 高 + 400ns 低 为 1）。连续发送多个灯的数据后，拉低超过 50μs 触发复位。

### Q7：ESP32 和 STM32 为什么分工？

> STM32 擅长实时控制、稳定可靠；ESP32 内置 Wi-Fi/蓝牙双核、算力更强，适合前置处理音频/网络协议。把"感知"放在 ESP32、把"执行"放在 STM32 是典型 IoT 分层思路，好处是解耦 + 两边都能发挥长处。

### Q8：如果让你优化，下一步你会做什么？

> 1. **闭环姿态控制**：用 MPU6050 检测倾倒并自动修正舵机角度。
> 2. **通信协议升级**：ASCII 改为二进制 + CRC，抗干扰更强。
> 3. **步态规划**：引入 S 曲线或三次样条插值，动作更自然。
> 4. **运动学外扩**：从二连杆扩展到 3 自由度髋关节，支持侧向步态。
> 5. **实时任务调度**：上 FreeRTOS，把通信、动作、状态灯各自放独立任务。

### Q9：遇到的最难的问题？怎么解决的？

> 舵机姿态标定。装配上的机械公差让每个舵机零位都不同，直接下发 90° 结果四条腿朝向都不一样。我通过手动微调每个舵机的起始角度，记录每个舵机的偏置值存到 `Px[]` 数组里，每次下发角度时自动叠加这个偏置，解决了姿态不一致问题。这让我理解了**软件补偿硬件公差**的思路。

### Q10：有哪些模块间通信？怎么保证可靠性？

> I2C（PCA9685、MPU6050）、UART（USART1 调试、USART2 蓝牙、UART4 ESP32）。可靠性靠几点：
> - I2C 带 ACK 机制，每字节有应答；
> - UART 加 IDLE 中断判断帧结束；
> - 应用层加指令校验（目前是简单的关键字匹配，可升级为 CRC）。

```c
#ifndef __PCA9685_H
#define __PCA9685_H

#include "stm32f4xx_hal.h"

/* PCA9685 I2C 地址（7位，HAL 库需要 << 1） */
#define PCA9685_ADDR            (0x40 << 1)

/* 关键寄存器地址 */
#define PCA9685_MODE1           0x00
#define PCA9685_MODE2           0x01
#define PCA9685_PRESCALE        0xFE
#define PCA9685_LED0_ON_L       0x06
#define PCA9685_LED0_ON_H       0x07
#define PCA9685_LED0_OFF_L      0x08
#define PCA9685_LED0_OFF_H      0x09

/* 内部振荡频率 */
#define PCA9685_OSC_FREQ        25000000UL

/* 通道数量 */
#define PCA9685_CHANNELS        16

/* 舵机角度范围对应 PWM 计数（基于 50Hz） */
#define SERVO_MIN_PULSE         102     /* 0.5ms */
#define SERVO_MAX_PULSE         512     /* 2.5ms */
#define SERVO_ANGLE_MAX         180.0f

/* API */

HAL_StatusTypeDef PCA9685_SetFreq(I2C_HandleTypeDef *hi2c, float freq_hz);
HAL_StatusTypeDef PCA9685_SetPWM(I2C_HandleTypeDef *hi2c, uint8_t ch, uint16_t on, uint16_t off);
```

**`pca9685.c`**

```c
#include "pca9685.h"
#include <math.h>

static HAL_StatusTypeDef PCA9685_WriteReg(I2C_HandleTypeDef *hi2c, uint8_t reg, uint8_t val) {
    return HAL_I2C_Mem_Write(hi2c, PCA9685_ADDR, reg, I2C_MEMADD_SIZE_8BIT, &val, 1, 100);
}

static HAL_StatusTypeDef PCA9685_ReadReg(I2C_HandleTypeDef *hi2c, uint8_t reg, uint8_t *val) {
    return HAL_I2C_Mem_Read(hi2c, PCA9685_ADDR, reg, I2C_MEMADD_SIZE_8BIT, val, 1, 100);
}

HAL_StatusTypeDef PCA9685_SetFreq(I2C_HandleTypeDef *hi2c, float freq_hz) {
    if (freq_hz < 24) freq_hz = 24;
    if (freq_hz > 1526) freq_hz = 1526;

    /* prescale = round(osc / (4096 * freq)) - 1 */
    float prescaleval = (PCA9685_OSC_FREQ / (4096.0f * freq_hz)) - 1.0f;
    uint8_t prescale = (uint8_t)(prescaleval + 0.5f);

    uint8_t oldmode;
    if (PCA9685_ReadReg(hi2c, PCA9685_MODE1, &oldmode) != HAL_OK) return HAL_ERROR;

    uint8_t newmode = (oldmode & 0x7F) | 0x10;           /* sleep 位置1，预分频才可写 */
    PCA9685_WriteReg(hi2c, PCA9685_MODE1, newmode);
    PCA9685_WriteReg(hi2c, PCA9685_PRESCALE, prescale);
    PCA9685_WriteReg(hi2c, PCA9685_MODE1, oldmode);
    HAL_Delay(5);
    PCA9685_WriteReg(hi2c, PCA9685_MODE1, oldmode | 0xA1); /* 自动增量 + 重启 */
    return HAL_OK;
}

HAL_StatusTypeDef PCA9685_Init(I2C_HandleTypeDef *hi2c, float freq_hz) {
    if (PCA9685_WriteReg(hi2c, PCA9685_MODE1, 0x00) != HAL_OK) return HAL_ERROR;
    return PCA9685_SetFreq(hi2c, freq_hz);
}

HAL_StatusTypeDef PCA9685_SetPWM(I2C_HandleTypeDef *hi2c, uint8_t ch, uint16_t on, uint16_t off) {
    uint8_t buf[4] = { on & 0xFF, (on >> 8) & 0x0F, off & 0xFF, (off >> 8) & 0x0F };
    uint8_t reg = PCA9685_LED0_ON_L + 4 * ch;
    return HAL_I2C_Mem_Write(hi2c, PCA9685_ADDR, reg, I2C_MEMADD_SIZE_8BIT, buf, 4, 100);
}

HAL_StatusTypeDef PCA9685_SetAngle(I2C_HandleTypeDef *hi2c, uint8_t ch, float angle) {
    if (angle < 0) angle = 0;
    if (angle > SERVO_ANGLE_MAX) angle = SERVO_ANGLE_MAX;
    uint16_t off = (uint16_t)(SERVO_MIN_PULSE + (SERVO_MAX_PULSE - SERVO_MIN_PULSE) * angle / SERVO_ANGLE_MAX);
    return PCA9685_SetPWM(hi2c, ch, 0, off);
}

HAL_StatusTypeDef PCA9685_SetAllAngle(I2C_HandleTypeDef *hi2c, float angle) {
    for (uint8_t i = 0; i < PCA9685_CHANNELS; i++) {
        if (PCA9685_SetAngle(hi2c, i, angle) != HAL_OK) return HAL_ERROR;
    }
    return HAL_OK;
}
```

**要点解读**：
- `PCA9685_SetFreq` 写预分频器前必须先把芯片置 sleep，否则寄存器只读。
- 地址宏 `<< 1` 是 HAL 库的规则（最低位给读/写位）。
- 用 `HAL_I2C_Mem_Write` 一次写 4 字节比 4 次单写高效（依赖 PCA9685 的自动增量 AI=1）。

```c
#define ACCEL_SENS_2G   16384.0f
#define GYRO_SENS_250   131.0f

HAL_StatusTypeDef MPU6050_Init(I2C_HandleTypeDef *hi2c) {
    uint8_t v;
    if (HAL_I2C_Mem_Read(hi2c, MPU6050_ADDR, MPU6050_WHO_AM_I, 1, &v, 1, 100) != HAL_OK) return HAL_ERROR;
    if (v != 0x68) return HAL_ERROR;

    uint8_t config[][2] = {
        {MPU6050_PWR_MGMT_1, 0x00},
        {MPU6050_SMPLRT_DIV, 0x07},
        {MPU6050_CONFIG,     0x06},
        {MPU6050_GYRO_CONFIG, 0x00},
        {MPU6050_ACCEL_CONFIG, 0x00},
    };
    for (int i = 0; i < 5; i++) {
        if (HAL_I2C_Mem_Write(hi2c, MPU6050_ADDR, config[i][0], 1, &config[i][1], 1, 100) != HAL_OK)
            return HAL_ERROR;
    }
    return HAL_OK;
}

HAL_StatusTypeDef MPU6050_ReadAll(I2C_HandleTypeDef *hi2c, MPU6050_Data_t *data) {
    uint8_t buf[14];
    if (HAL_I2C_Mem_Read(hi2c, MPU6050_ADDR, MPU6050_ACCEL_XOUT_H, 1, buf, 14, 100) != HAL_OK)
        return HAL_ERROR;

    int16_t ax = (buf[0] << 8) | buf[1];
    int16_t ay = (buf[2] << 8) | buf[3];
    int16_t az = (buf[4] << 8) | buf[5];
    int16_t tr = (buf[6] << 8) | buf[7];
    int16_t gx = (buf[8] << 8) | buf[9];
    int16_t gy = (buf[10] << 8) | buf[11];
    int16_t gz = (buf[12] << 8) | buf[13];

    data->ax = ax / ACCEL_SENS_2G;
    data->ay = ay / ACCEL_SENS_2G;
    data->az = az / ACCEL_SENS_2G;
    data->gx = gx / GYRO_SENS_250;
    data->gy = gy / GYRO_SENS_250;
    data->gz = gz / GYRO_SENS_250;
    data->temp = tr / 340.0f + 36.53f;
    return HAL_OK;
}

uint8_t MPU6050_Check(I2C_HandleTypeDef *hi2c) {
    uint8_t v = 0;
    HAL_I2C_Mem_Read(hi2c, MPU6050_ADDR, MPU6050_WHO_AM_I, 1, &v, 1, 100);
    return v;
}
```

### 11.3 运动学逆解完整代码

**`kinematics.h`**

```c
#ifndef __KINEMATICS_H
#define __KINEMATICS_H

#define LEG_L1   50.0f
#define LEG_L2   60.0f

int InverseResolve(float x, float y, float *a1, float *a2);

#endif
```

**`kinematics.c`**

```c
#include "kinematics.h"
#include <math.h>

#define PI 3.14159265358979f
#define RAD2DEG(x)  ((x) * 180.0f / PI)

int InverseResolve(float x, float y, float *a1, float *a2) {
    float d2 = x * x + y * y;
    float d  = sqrtf(d2);

    float max_reach = LEG_L1 + LEG_L2;
    float min_reach = fabsf(LEG_L1 - LEG_L2);
    if (d > max_reach - 0.1f || d < min_reach + 0.1f) {
        return -1;
    }

    float cos_knee = (LEG_L1 * LEG_L1 + LEG_L2 * LEG_L2 - d2) / (2.0f * LEG_L1 * LEG_L2);
    if (cos_knee > 1.0f)  cos_knee = 1.0f;
    if (cos_knee < -1.0f) cos_knee = -1.0f;
    float knee = acosf(cos_knee);
    *a2 = RAD2DEG(PI - knee);

    float alpha = atan2f(x, y);
    float cos_beta = (LEG_L1 * LEG_L1 + d2 - LEG_L2 * LEG_L2) / (2.0f * LEG_L1 * d);
    if (cos_beta > 1.0f)  cos_beta = 1.0f;
    if (cos_beta < -1.0f) cos_beta = -1.0f;
    float beta = acosf(cos_beta);
    *a1 = RAD2DEG(alpha - beta) + 90.0f;

    if (*a1 < 0 || *a1 > 180 || *a2 < 0 || *a2 > 180) return -1;
    return 0;
}
```

### 11.4 舵机平滑过渡（带状态保存）

```c
#define NUM_SERVOS  16

static float g_current_angle[NUM_SERVOS] = {0};
static const float g_step[NUM_SERVOS]    = {
    1.0f, 1.0f, 1.0f, 1.0f,
    1.0f, 1.0f, 1.0f, 1.0f,
    1.0f, 1.0f, 1.0f, 1.0f,
    1.0f, 1.0f, 1.0f, 1.0f
};

int AngleMoveStep(I2C_HandleTypeDef *hi2c, uint8_t ch, float target) {
    float current = g_current_angle[ch];
    float diff = target - current;
    if (fabsf(diff) <= g_step[ch]) {
        PCA9685_SetAngle(hi2c, ch, target);
        g_current_angle[ch] = target;
        return 1;
    }
    current += (diff > 0) ? g_step[ch] : -g_step[ch];
    PCA9685_SetAngle(hi2c, ch, current);
    g_current_angle[ch] = current;
    return 0;
}

void AngleMoveAll(I2C_HandleTypeDef *hi2c, float targets[NUM_SERVOS], uint32_t tick_ms) {
    while (1) {
        int all_done = 1;
        for (int i = 0; i < NUM_SERVOS; i++) {
            if (!AngleMoveStep(hi2c, i, targets[i])) all_done = 0;
        }
        if (all_done) break;
        HAL_Delay(tick_ms);
    }
}
```

### 11.5 动作控制（以 DogRun 为例）

```c
typedef struct {
    float x_fl, y_fl;
    float x_fr, y_fr;
    float x_bl, y_bl;
    float x_br, y_br;
    uint16_t delay_ms;
} ActionFrame_t;

static const ActionFrame_t Trot_Forward[] = {
    { 20, 60, -20, 60,  -20, 60,  20, 60, 100 },
    { 30, 50, -20, 60,  -20, 60,  30, 50,  80 },
    { 30, 60, -20, 60,  -20, 60,  30, 60,  80 },
    {-20, 60,  30, 50,   30, 50, -20, 60,  80 },
    {-20, 60,  30, 60,   30, 60, -20, 60,  80 },
    { 0 }
};

static const uint8_t LEG_SERVO_MAP[4][2] = {
    {0, 1}, {2, 3}, {4, 5}, {6, 7}
};

static const float PX[8] = {0, 0, 0, 0, 0, 0, 0, 0};

void SetLegAngle(I2C_HandleTypeDef *hi2c, uint8_t leg, float a1, float a2) {
    uint8_t c1 = LEG_SERVO_MAP[leg][0];
    uint8_t c2 = LEG_SERVO_MAP[leg][1];
    PCA9685_SetAngle(hi2c, c1, a1 + PX[c1]);
    PCA9685_SetAngle(hi2c, c2, a2 + PX[c2]);
}

int ExecFrame(I2C_HandleTypeDef *hi2c, const ActionFrame_t *f) {
    float a1, a2;
    if (InverseResolve(f->x_fl, f->y_fl, &a1, &a2) < 0) return -1;
    SetLegAngle(hi2c, 0, a1, a2);
    if (InverseResolve(f->x_fr, f->y_fr, &a1, &a2) < 0) return -1;
    SetLegAngle(hi2c, 1, a1, a2);
    if (InverseResolve(f->x_bl, f->y_bl, &a1, &a2) < 0) return -1;
    SetLegAngle(hi2c, 2, a1, a2);
    if (InverseResolve(f->x_br, f->y_br, &a1, &a2) < 0) return -1;
    SetLegAngle(hi2c, 3, a1, a2);
    HAL_Delay(f->delay_ms);
    return 0;
}

void DogRun(I2C_HandleTypeDef *hi2c) {
    for (int i = 0; Trot_Forward[i].delay_ms != 0; i++) {
        ExecFrame(hi2c, &Trot_Forward[i]);
    }
}
```

```c
    {"RIGHT",   DogRight},
    {"SQUAT",   DogSquat},
    {"WAVE",    DogWave},
    {NULL, NULL}

void ParseAndExec(I2C_HandleTypeDef *hi2c, const char *str) {
    for (int i = 0; g_cmd_table[i].keyword != NULL; i++) {
        if (strstr(str, g_cmd_table[i].keyword)) {
            g_cmd_table[i].func(hi2c);
            printf("[CMD] %s executed\r\n", g_cmd_table[i].keyword);
            return;
        }
    }
    printf("[CMD] unknown: %s\r\n", str);
}
```

## 12. 姿态闭环（PID + MPU6050 互补滤波）

### 12.1 原理

舵机角度是开环的，推一下就摔。接入 MPU6050 估算 Pitch / Roll，通过 PID 实时调整腿高，实现"被推不倒"。

### 12.2 互补滤波估算欧拉角

```c
typedef struct {
    float pitch, roll;
    float dt;
    float alpha;
    uint32_t last_tick;
} AttitudeFilter_t;

void Attitude_Init(AttitudeFilter_t *af) {
    af->pitch = af->roll = 0;
    af->alpha = 0.98f;
    af->last_tick = HAL_GetTick();
}

void Attitude_Update(AttitudeFilter_t *af, MPU6050_Data_t *d) {
    uint32_t now = HAL_GetTick();
    af->dt = (now - af->last_tick) / 1000.0f;
    af->last_tick = now;

    float acc_roll  = atan2f(d->ay, d->az) * 57.2958f;
    float acc_pitch = atan2f(-d->ax, sqrtf(d->ay*d->ay + d->az*d->az)) * 57.2958f;

    float gyro_roll  = af->roll  + d->gx * af->dt;
    float gyro_pitch = af->pitch + d->gy * af->dt;

    af->roll  = af->alpha * gyro_roll  + (1 - af->alpha) * acc_roll;
    af->pitch = af->alpha * gyro_pitch + (1 - af->alpha) * acc_pitch;
}
```

### 12.3 增量式 PID

```c
typedef struct {
    float kp, ki, kd;
    float err_prev, err_prev2;
    float out_prev;
    float out_max, out_min;
} PID_t;

float PID_Inc(PID_t *p, float err) {
    float du = p->kp * (err - p->err_prev)
             + p->ki * err
             + p->kd * (err - 2*p->err_prev + p->err_prev2);
    p->err_prev2 = p->err_prev;
    p->err_prev  = err;
    p->out_prev += du;
    if (p->out_prev > p->out_max) p->out_prev = p->out_max;
    if (p->out_prev < p->out_min) p->out_prev = p->out_min;
    return p->out_prev;
}
```

### 12.4 姿态自稳循环

```c
AttitudeFilter_t g_att;
PID_t g_pid_pitch = {0.5f, 0.02f, 0.1f, 0, 0, 0, +20, -20};
PID_t g_pid_roll  = {0.5f, 0.02f, 0.1f, 0, 0, 0, +20, -20};

void StabilizeTask(void) {
    MPU6050_Data_t d;
    MPU6050_ReadAll(&hi2c1, &d);
    Attitude_Update(&g_att, &d);

    float dh_pitch = PID_Inc(&g_pid_pitch, 0 - g_att.pitch);
    float dh_roll  = PID_Inc(&g_pid_roll,  0 - g_att.roll);

    float h = 60.0f;
    float legs_y[4] = {
        h + dh_pitch - dh_roll,
        h + dh_pitch + dh_roll,
        h - dh_pitch - dh_roll,
        h - dh_pitch + dh_roll
    };
    for (int i = 0; i < 4; i++) {
        float a1, a2;
        if (InverseResolve(0, legs_y[i], &a1, &a2) == 0) {
            SetLegAngle(&hi2c1, i, a1, a2);
        }
    }
}
```

**调用频率**：10~50Hz。
**简历话术**：基于 MPU6050 + 互补滤波 + 增量式 PID 实现机身姿态闭环自稳，机器狗被外力推动后可自动恢复水平。

## 13. FreeRTOS 多任务重构

### 13.1 任务划分

| 任务 | 优先级 | 周期 | 职责 |
|------|--------|------|------|
| `StabilizeTask` | 3（高） | 20ms | 姿态闭环 |
| `ActionTask` | 2 | 事件 | 执行动作序列 |
| `UartRxTask` | 2 | 事件 | 解析串口指令 |
| `LedTask` | 1 | 50ms | 状态灯刷新 |
| `LogTask` | 1 | 100ms | 调试日志 |

### 13.2 关键代码

```c
#include "FreeRTOS.h"
#include "task.h"
#include "queue.h"
#include "semphr.h"

QueueHandle_t xCmdQueue;
SemaphoreHandle_t xI2CMutex;

void vStabilizeTask(void *arg) {
    const TickType_t period = pdMS_TO_TICKS(20);
    TickType_t last = xTaskGetTickCount();
    for (;;) {
        xSemaphoreTake(xI2CMutex, portMAX_DELAY);
        StabilizeTask();
        xSemaphoreGive(xI2CMutex);
        vTaskDelayUntil(&last, period);
    }
}

void vActionTask(void *arg) {
    uint8_t cmd;
    for (;;) {
        if (xQueueReceive(xCmdQueue, &cmd, portMAX_DELAY) == pdTRUE) {
            xSemaphoreTake(xI2CMutex, portMAX_DELAY);
            switch (cmd) {
                case 1: DogRun(&hi2c1);   break;
                case 2: DogBack(&hi2c1);  break;
                case 3: DogLeft(&hi2c1);  break;
                case 4: DogRight(&hi2c1); break;
            }
            xSemaphoreGive(xI2CMutex);
        }
    }
}

int main(void) {
    /* HAL init ... */
    xCmdQueue = xQueueCreate(8, sizeof(uint8_t));
    xI2CMutex = xSemaphoreCreateMutex();
    xTaskCreate(vStabilizeTask, "STAB", 512, NULL, 3, NULL);
    xTaskCreate(vActionTask,    "ACT",  512, NULL, 2, NULL);
    xTaskCreate(vUartRxTask,    "UART", 256, NULL, 2, NULL);
    xTaskCreate(vLedTask,       "LED",  128, NULL, 1, NULL);
    vTaskStartScheduler();
    while (1);
}
```

**简历话术**：基于 FreeRTOS 将系统重构为多任务并行架构，使用互斥量保护 I2C 总线，队列解耦通信与执行，系统响应延迟低于 20ms。

|-------|------|---------|

### 14.2 ESP32 核心代码

```cpp
#include <WiFi.h>
#include <PubSubClient.h>

WiFiClient espClient;
PubSubClient client(espClient);

void mqttCallback(char *topic, byte *payload, unsigned int length) {
    String msg = "";
    for (int i = 0; i < length; i++) msg += (char)payload[i];
    int p = msg.indexOf("cmd");
    if (p < 0) return;
    int q1 = msg.indexOf('"', p + 5);
    int q2 = msg.indexOf('"', q1 + 1);
    String cmd = msg.substring(q1 + 1, q2);
    Serial2.print(cmd + "\n");
}

void setup() {
    Serial.begin(115200);
    Serial2.begin(115200, SERIAL_8N1, 16, 17);
    WiFi.begin("YOUR_SSID", "YOUR_PASS");
    while (WiFi.status() != WL_CONNECTED) delay(500);
    client.setServer("broker.emqx.io", 1883);
    client.setCallback(mqttCallback);
}

void loop() {
    if (!client.connected()) {
        if (client.connect("dog-0001")) {
            client.subscribe("dog/0001/cmd");
        }
    }
    client.loop();

    static unsigned long last = 0;
    if (millis() - last > 1000) {
        if (Serial2.available()) {
            String state = Serial2.readStringUntil('\n');
            client.publish("dog/0001/state", state.c_str());
        }
        last = millis();
    }
}
```

**简历话术**：基于 MQTT 协议实现手机 APP 与机器狗的端到端远程通信，支持云端指令下发与状态实时上报。

## 16. S 曲线步态插值

```c
/* 三次 Hermite 平滑：t ∈ [0,1] 映射到 [0,1] 但端点导数为 0 */
float SCurve(float t) {
    return t * t * (3.0f - 2.0f * t);
}

void AngleMoveSCurve(I2C_HandleTypeDef *hi2c, uint8_t ch,
                     float target, int steps, uint16_t dt_ms) {
    float start = g_current_angle[ch];
    float span  = target - start;
    for (int i = 1; i <= steps; i++) {
        float t = (float)i / steps;
        float progress = SCurve(t);
        float ang = start + span * progress;
        PCA9685_SetAngle(hi2c, ch, ang);
        HAL_Delay(dt_ms);
    }
    g_current_angle[ch] = target;
}
```

**简历话术**：用三次 Hermite 平滑曲线替代线性插值，加速度连续，动作更接近生物运动。

## 22. 面试扩展 Q&A（衍生功能版）

### Q11：为什么用互补滤波而不用卡尔曼？

> 卡尔曼滤波效果更好但实现复杂、计算开销大；互补滤波只需一个系数 α，在 STM32F407 上运行成本低、调参简单、对姿态平衡已经够用。如果精度要求高可以升级到 MPU6050 内置 DMP 或卡尔曼滤波。

### Q12：PID 参数怎么调？

> 先只调 P：从小到大加，直到出现明显振荡但能跟踪目标；然后减半。再调 D 抑制振荡。最后加一点点 I 消除静差。工程上常用"试凑法" + "临界比例度法"。

### Q13：为什么用增量式 PID？

> 增量式输出的是"变化量"而不是绝对值，切换模式时不会突变，避免机械冲击；另外增量式不需要对误差做积分累加，抗积分饱和能力强。

### Q14：FreeRTOS 里 I2C 为什么要加互斥量？

> I2C 操作涉及 START → 地址 → 数据 → STOP 的原子序列，如果两个任务同时读 PCA9685 和 MPU6050，中断会交叉破坏时序，导致总线死锁。互斥量保证同时只有一个任务操作总线。

### Q15：MQTT 为什么比 HTTP 更适合 IoT？

> MQTT 是发布/订阅模型，长连接、消息小（最少 2 字节头）、支持 QoS 保证可达、遗嘱消息处理掉线。HTTP 每次请求都要握手、头开销大、不适合资源受限设备和高实时性场景。
